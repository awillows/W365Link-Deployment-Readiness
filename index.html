Windows 365 Link — Deployment Readiness Guide

This interactive guide walks administrators through every prerequisite and configuration step needed to successfully deploy Windows 365 Link devices. Each section includes validation checklists and highlights areas worth confirming before you begin.

Key Guidance

Why this guide?

Windows 365 Link is a purpose-built Cloud PC device that relies on configurations across Microsoft Entra ID, Microsoft Intune, and Windows 365. Confirming these settings ahead of time helps ensure a smooth onboarding experience for your users.

📊 Key Areas to Verify
We recommend paying particular attention to these areas when preparing for deployment:
  1. SSO enabled on Cloud PCs — Windows 365 Link connects using Entra ID SSO, so provisioning policies need this enabled
  2. SSO consent prompt suppressed — Suppressing the consent prompt ensures a seamless connection experience
  3. Conditional Access alignment — The two-stage authentication model may need Conditional Access policies reviewed
  4. Enrollment restrictions reviewed — Link devices initially register as "Unknown/Personal", so existing restrictions should be checked
  5. MDM User Scope configured — Enables automatic Intune enrollment after Entra ID join

How to use this guide

Work through each section in order using the sidebar navigation. Check off each item as you confirm or complete it in your environment. The Deployment Readiness score in the sidebar tracks your overall progress. Sections marked with Required must be completed. Sections marked Recommended are strongly advised.

Step 1: Licensing

Windows 365 Link requires specific licenses for both the device management plane and the Cloud PC service. Verifying these are in place helps ensure enrollment and connectivity work as expected.

💡 Good to know
Entra ID Premium license: This license enables automatic MDM enrollment. Without it, a device may join Entra ID but not appear in Intune. Confirming this license is in place beforehand avoids a common configuration gap.

Required licenses

  • Windows 365 Enterprise, Frontline, or Business license
    Provides the Cloud PC that Link connects to. Windows 365 Business also requires Intune.
  • Microsoft Intune license
    Required for device management. Included with some M365 plans (E3/E5, Business Premium).
  • Microsoft Entra ID Premium license (P1 or P2)
    Required for automatic MDM enrollment during Entra join. Without it, devices join Entra but do NOT enroll in Intune.
  • Cloud PC is provisioned for each target user
    Users must have an active Cloud PC before they can connect from a Link device.
▶ GCC / Government customers

Windows 365 Link can only be used with the Windows 365 Enterprise/Frontline for FedRAMP GCC offering. Windows 365 Business is not supported for GCC. See What is Windows 365 Government?

Step 2: Microsoft Entra ID — Device Join

Windows 365 Link must be Microsoft Entra joined. During the Out of Box Experience (OOBE), the first user signs in and the device joins their tenant. Users must have permission to join devices.

⚠️ Worth checking
If "Users may join devices to Microsoft Entra" is set to "Selected", make sure your target users are in the selected group. If it's set to "None", Link devices won't be able to join. Confirming this setting avoids an unexpected error during OOBE.

Configuration checklist

  • Entra ID device join is enabled for target users
    Entra admin center → Identity → Devices → Overview → Device Settings → "Users may join devices to Microsoft Entra" set to All or Selected.
  • "Maximum number of devices per user" is adequate
    Default is 50. If using a DEM account for bulk onboarding, this limit applies per DEM user. Consider increasing it.
  • If "Selected" — confirmed target user groups include all Link device users
    The selected groups must be Entra ID groups containing user objects (not device objects).
💡 Good to know
Device join limit: The default is 50 devices per user. If a Device Enrollment Manager (DEM) account is onboarding many Link devices, it's worth confirming this limit has room. You can increase it in Entra ID device settings if needed.

✓ How to verify

Entra admin center → Identity → Devices → Overview → Device Settings

Confirm that the "Users may join devices to Microsoft Entra" setting is either All or Selected with the correct groups. Check the device limit value to accommodate expected device counts.

Step 3: Intune Automatic Enrollment

After a Windows 365 Link device joins Entra ID, it needs to automatically enroll in Intune via the MDM User Scope setting. This is an important step to confirm — if MDM auto-enrollment isn't enabled, the device will join Entra ID but won't appear in Intune.

💡 Important configuration
If MDM user scope is set to "None", or the user isn't in scope, the device will complete OOBE and appear in Entra ID but won't be enrolled in Intune. Setting this to "All" or including your target users in a "Some" scope ensures devices are managed from day one.

Configuration checklist

  • MDM User Scope is set to "All" or "Some"
    Entra admin center → Settings → Mobility → Microsoft Intune → MDM user scope.
  • If "Some" — target users are in the selected groups
    Ensure the groups contain the user objects who will be onboarding Link devices.
  • No conflicting MDM applications on the Mobility page
    If there are multiple applications listed, confirm only Microsoft Intune has MDM user scope enabled. Having conflicting MDM apps causes enrollment to go to the wrong MDM.
  • MDM Authority is set to "Microsoft Intune"
    Intune admin center → Tenant administration → Tenant status → Tenant details.

✓ How to verify

Entra admin center → Show more → Settings → Mobility (MDM and WIP) → Microsoft Intune

Confirm MDM user scope is set to All or Some (with correct groups). Then verify in Intune admin center that MDM authority shows "Microsoft Intune".

Step 4: Enrollment Restrictions

Windows 365 Link devices initially appear as "Unknown" ownership during enrollment. If your environment has Intune platform restrictions that block personally owned or unknown Windows devices, these may also apply to Link devices. It's worth reviewing your restrictions before deployment.

💡 Important to verify
Many organizations have restrictions blocking personal Windows devices. Because Link devices are "Unknown" at enrollment time (before they become "Corporate-owned" after join), they can be affected by these restrictions. Reviewing and adjusting your restriction policies ensures Link devices enroll smoothly.

Do you have restrictions blocking personal Windows devices?

Does your environment have an Intune platform restriction that blocks personally owned Windows devices?
Action needed. You must use one of these three methods to allow Link devices through:
  1. Corporate Identifiers — Pre-upload serial numbers so devices are recognized as corporate before restrictions apply
  2. OS SKU Filter — Create a higher-priority policy with an Intune filter for Windows CPC (210) to allow Link devices
  3. Device Enrollment Manager — DEM accounts bypass all platform restrictions
Likely OK. If you don't block personal devices, Link devices should enroll successfully. However, we still recommend creating an Intune filter for Windows 365 Link devices for future policy targeting.

Configuration checklist

  • Reviewed existing Windows platform enrollment restrictions
    Intune admin center → Devices → Enrollment → Windows → Device platform restriction → Windows restrictions.
  • If blocking personal devices: implemented bypass method (Corporate IDs, SKU filter, or DEM)
    See the decision tree above. At least one method must be in place before OOBE.
  • Created an Intune filter for Windows 365 Link devices
    Intune → Tenant admin → Filters → Create → Platform: Windows 10 and later → Property: operatingSystemSKU → Equals → WCPC (Windows CPC (210)).
▶ How to upload Corporate Identifiers

Create a CSV file with: Microsoft Corporation,Windows 365 Link,<SerialNumber>

Serial numbers are on the bottom of the device and on the original packaging.

Intune admin center → Devices → Enrollment → Corporate device identifiers → Add → Upload CSV file

Select identifier type: Manufacturer, model, and serial number (Windows only). Wait for validation, then select Add.

Step 5: Single Sign-On (SSO) Configuration

Windows 365 Link connects to Cloud PCs using Entra ID SSO, so this must be enabled on provisioning policies. Additionally, suppressing the SSO consent prompt provides the best user experience. These two settings are key to a seamless connection.

💡 Important — SSO is required for Link
If SSO is not enabled on the Cloud PC's provisioning policy, the connection won't succeed. The fix is straightforward — enable SSO in the provisioning policy settings and the change takes effect on the next connection.
⚠️ SSO Consent Prompt
Even with SSO enabled, users may be prompted for consent the first time and periodically thereafter. Link uses a non-interactive SSO connection that cannot display this prompt. Suppressing the consent prompt ensures users connect seamlessly every time. See the checklist below for how to configure this.

Configuration checklist

  • SSO is enabled on ALL Cloud PC provisioning policies used by Link users
    Intune → Devices → Windows 365 → Provisioning policies → Edit → Enable single sign-on.
  • Verified SSO status on existing Cloud PCs
    Intune → Devices → Windows 365 → All Cloud PCs → Add column "Using single sign-on" → Confirm all target PCs show "Yes".
  • Created a dynamic device group for all Cloud PCs
    Required for SSO consent suppression. Membership rule should target Cloud PC device objects.
  • Enabled Entra authentication for RDP on the SSO service principal
    Configure the "Windows Cloud Login" service principal.
  • Added Cloud PC group to the Service Principal target (consent suppression)
    This hides the consent dialog so Link devices can connect without user interaction.
  • If using Conditional Access with Cloud PCs — included SSO Cloud App in CA policy target resources
    The SSO service principal must be in the CA policy target resources alongside Windows 365 and Azure Virtual Desktop.
ℹ️ Note — December 2025 Update (26100.7462+)
Devices on build 26100.7462 or later can now show interactive SSO consent prompts. However, Microsoft still strongly recommends suppressing the prompt to avoid a degraded user experience.

Step 6: Conditional Access Policies

Windows 365 Link authentication happens in two stages, and each stage triggers different Conditional Access policies. It's important to ensure these stages are aligned so users have a seamless connection experience.

📊 Understanding the two-stage authentication
StageTypeCA Policy TriggerCan prompt user?
1. Sign-inInteractiveUser actions: "Register or join devices"✔ Yes
2. Connect to Cloud PCNon-interactive SSOResources: Windows 365, Azure Virtual Desktop, Windows Cloud Login✘ No

If Stage 2 requires MFA but Stage 1 didn't satisfy it (because there's no matching user-action policy), devices on builds before 26100.7462 may not connect. Since build 26100.7462, the device can prompt for additional authentication — but Microsoft still highly recommends creating a user-action policy as a best practice.

Configuration checklist

  • Used the "What If" tool to identify CA policies applied to Windows 365 resources
    Entra admin center → Protection → Conditional Access → What If. Test with Windows 365 (0af06dc6), Azure Virtual Desktop (9cdead84), Windows Cloud Login (270efc09).
  • Created a CA policy for user action "Register or join devices" with matching controls Recommended
    Since build 26100.7462, this is no longer mandatory — but Microsoft still highly recommends it. The Grant controls on this policy should match (or be stronger than) the Resource policies found with What If. Exclude emergency/break-glass accounts.
  • Tested the new policy in Report-Only mode first
    Set "Enable policy" to Report-only, validate via sign-in logs, then switch to On.
  • Confirmed no unsupported Grant or Session controls on user-action policy
    Device compliance, custom controls, and sign-in frequency CANNOT be used with "Register or join devices" user actions. These will cause failures.
▶ Unsupported controls for "Register or join devices" policies

The following controls are NOT supported when the target is a user action (Register or join devices):

Control typeUnsupported controlImpact
GrantRequire device to be marked as compliantCan't evaluate compliance during registration
GrantCustom controls (legacy)Not available for user actions
SessionSign-in frequencyNot applicable to device registration

Step 7: Authentication Methods Recommended

Windows 365 Link supports two sign-in methods: Web sign-in (enabled by default) and FIDO2 security keys (requires configuration). Microsoft strongly recommends enabling FIDO2 for the best security and user experience.

⚠️ Certificate-based authentication (CBA) is NOT supported
If your organization relies on Microsoft Entra certificate-based authentication, it cannot be used for Web sign-in on Windows 365 Link. Plan alternative authentication methods before deployment.
⚠️ Third-party IdP custom controls — enrollment compatibility
If your third-party identity provider uses custom controls to integrate with Entra ID, be aware these do not support Intune device enrollment or Entra ID join. Use Entra's external authentication method instead of custom controls.

Configuration checklist

  • FIDO2 security key sign-in is enabled in Entra ID
    Entra admin center → Protection → Authentication methods → FIDO2 security key → Enable for target users. Without this, the security key icon won't appear.
  • Confirmed Link sign-in screen shows both icons (globe + security key)
    If only the globe icon appears, FIDO2 is not configured for the tenant/user.
  • If using third-party IdP — verified no custom controls are in use
    Custom controls can't satisfy MFA, can't support enrollment, and can't support Entra join.
▶ Web sign-in authentication options

Web sign-in supports the following Entra ID authentication methods (no device configuration needed):

  • Username and password
  • Phone sign-in with Microsoft Authenticator
  • FIDO2 security key (via web sign-in flow)
  • Passkey on another device via cross-device authentication

Step 8: Microsoft Teams & Media Optimizations Recommended

Windows 365 Link uses the VDI 2.0 solution for Teams for media optimizations (audio/video). The VDI component is preinstalled on the device, but the Teams admin must ensure users are in scope for the new VDI policy.

💡 Good to know
Teams media optimizations: If the Teams VDI 2.0 policy doesn't include Link users, media optimizations won't be active. Audio and video calls will still work, but routing through the Cloud PC without local optimization may affect call quality. Including Link users in the VDI 2.0 policy ensures the best experience.

Configuration checklist

  • Verified Teams VDI 2.0 policy includes Link users
    Check the Microsoft Teams PowerShell policy for optimization. Ensure users signing in to Link devices are in scope for the VDI policy.
  • Confirmed multimedia redirection is working in a test call
    Make a test call from a Link device and verify audio/video quality. Local redirection should handle media.
▶ More about VDI 2.0 for Teams

The VDI 2.0 solution for Teams is the latest optimization approach that handles media locally on the endpoint for improved call quality. Windows 365 Link includes this component in its operating system — no installation needed. However, the server-side Teams policy must be configured to allow VDI optimization for these users.

For full details, see New VDI solution for Teams.

Step 9: Network Requirements

Windows 365 Link has the same network requirements as Azure Virtual Desktop end user devices. Confirming that your firewall and proxy configurations allow the necessary connections helps ensure devices can connect without issues.

ℹ️ Key network requirements
All traffic from Windows 365 Link to Windows 365 services uses TLS 1.2 or greater. The device needs outbound HTTPS (443) access to Microsoft Entra ID, Intune, and Windows 365/AVD service endpoints.

Configuration checklist

  • Reviewed Azure Virtual Desktop required FQDNs/endpoints for end user devices
    See AVD Required FQDN and endpoints.
  • Firewall/proxy allows outbound HTTPS to Microsoft Entra, Intune, and W365 services
    Ensure *.microsoft.com, login.microsoftonline.com, *.windows.net, and related endpoints are not blocked.
  • WiFi or Ethernet connectivity available at deployment locations
    Link supports WiFi 6E and 1 Gbps Ethernet. Captive portal WiFi is supported on build 26100.6584+.
  • TLS inspection (if any) does not break Microsoft service connections
    TLS inspection/SSL decryption can interfere with certificate pinning used by Link. Test connectivity before bulk deployment.

Step 10: Onboarding Strategy

Windows 365 Link supports two onboarding approaches: Admin-driven (using a Device Enrollment Manager) and User-driven (user completes OOBE themselves). Choose the right model based on your deployment scenario.

Which onboarding model is right for you?

How will your Link devices primarily be used?
Recommended: Admin-driven with DEM account.
  • Create a dedicated DEM account with required licenses (Entra Premium + Intune + Windows)
  • DEM bypasses enrollment restrictions and device limits
  • DEM is set as primary user, but any org user can sign in
  • Increase "Maximum devices per user" for the DEM account
  • Provision a test Cloud PC for the DEM account to validate connectivity
Recommended: User-driven onboarding.
  • Ship devices directly to users
  • Ensure each user has correct licenses and permissions
  • User becomes primary user of the device
  • User must have a Cloud PC provisioned WITH SSO and consent already granted
Use both methods as appropriate.
  • Admin-driven for shared/hot-desk devices
  • User-driven for personal devices shipped to users
  • Ensure both paths are tested before bulk deployment

Post-onboarding checklist

  • Decided on onboarding model (Admin, User, or mixed)
    Use the decision tree above to guide your choice.
  • If Admin-driven: DEM account created, licensed, and tested
    Verify the DEM can complete OOBE successfully and connect to a test Cloud PC.
  • First device onboarded, updated, and verified in Intune
    After OOBE, disconnect → restart → update → shut down. Then verify the device appears in both Entra and Intune.
  • Recommended post-onboard configs applied (time zone, screen timeout)
    Auto Detect Local Time Zone is strongly recommended. Screen timeout can be customized to your needs.

Step 11: End-to-End Validation

Before bulk deployment, we recommend performing an end-to-end validation with at least one Link device. This confirms all the pieces are working together and gives you confidence before scaling out.

Validation checklist

  • Device completes OOBE — joins Entra ID and enrolls in Intune
    Verify the device appears in both the Entra admin center (Devices) and Intune admin center (Devices → All devices).
  • User connects to Cloud PC without errors
    No SSO errors, no "interactive window" errors, no consent prompts.
  • Second user can sign in to the same device and reach their own Cloud PC
    Tests the shared device scenario. First user's data must not be accessible.
  • Lock, sign-out, and disconnect all work correctly
    Test Win+L, Ctrl+Alt+Del → Sign out, and Start → Power → Disconnect. Each should return to sign-in screen.
  • Teams call works with audio/video (media optimization)
    Make a test call and verify good quality — confirming VDI 2.0 is active.
  • Device receives updates when in standby/sleep mode
    Leave powered on overnight. OS updates occur automatically during off hours.
  • Intune compliance shows device as compliant
    Check compliance status: BitLocker, Secure Boot, and code integrity are enabled by default.

Troubleshooting quick reference

▶ "Your Cloud PC does not support Entra ID single sign-on"

Cause: SSO not enabled on Cloud PC provisioning policy.

Fix: Edit the provisioning policy → Enable SSO → Apply to all affected Cloud PCs. See Step 5.

▶ "Failed to open a Microsoft Entra ID credential prompt"

Cause: SSO consent prompt not suppressed. Happens first time, after reprovisioning, and every 30 days.

Workaround: Connect from another device/web browser first to grant consent.

Fix: Suppress the consent prompt on the SSO service principals. See Step 5.

▶ "An interactive window could not be shown"

Cause: Conditional Access policy mismatch between sign-in (interactive) and connection (non-interactive) stages.

Fix: Create a CA policy targeting user action "Register or join devices" with matching Grant controls. See Step 6.

▶ Device doesn't appear in Intune after OOBE

Cause 1: MDM User Scope not set, or user not in scope. See Step 3.

Cause 2: Enrollment restriction blocking unknown/personal devices. See Step 4.

Cause 3: Missing Entra ID Premium license for the onboarding user. See Step 1.

▶ OOBE fails during device join

Cause 1: User not permitted to join devices to Entra ID. See Step 2.

Cause 2: Maximum device limit reached for the user. Increase the limit in Entra device settings.

Cause 3: Network connectivity issue — ensure device can reach Microsoft endpoints.
